# 2. API Philosophy

The API follows these principles.

## REST First

Resources are represented using nouns.

Correct

GET /jobs

POST /jobs

Incorrect

GET /getJobs

POST /createJob

---

## Predictable

Every endpoint follows identical conventions.

Developers should never need to guess request formats.

---

## Consistent

Responses always use the same structure.

Errors always use the same structure.

Authentication always behaves identically.

---

## Stateless

Every request contains everything required to execute it.

No server-side session state is maintained.

Authentication relies on Clerk JWT tokens.

---

## Secure

Every request is authenticated.

Every business request is scoped to an organization.

Authorization is enforced before business logic executes.

# 39. Design Decisions

Authentication remains fully delegated to Clerk.

Organization ID is never accepted from the frontend.

All candidate operations are organization-scoped.

Resume upload and resume analysis are separate endpoints because analysis is asynchronous.

Job matching is generated dynamically rather than stored permanently.

Applications are the central entity in the hiring workflow.

# 54. API Security Guidelines

Every protected endpoint must:

✓ Verify Clerk JWT

✓ Resolve organization membership

✓ Enforce role permissions

✓ Validate request data

✓ Return standardized responses

The backend must never trust:

- organization_id supplied by the client
- role supplied by the client
- user_id supplied by the client (unless explicitly required and authorized)

Authorization decisions are derived from the authenticated user context.
